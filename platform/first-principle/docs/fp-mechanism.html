First Principle 运作机制 v3 · 2026-04-24 · 吸收 6 天结构变化

一句话定位：First Principle（下称 FP）是 Supermatrix 多 session 运行时里 三份 Principles + 四套分类模板 + 治理规则 的 唯一事实源（SSoT）。 它以"仓库即数据库、symlink 即分发、scheduler 即心跳、soft-distribute 即协作"这四条原则运转。

七问七答（用户问题速查）

整个架构怎么运转？
FP 的工作区是 SSoT。三份 Principles 通过 symlink 被每个 session 挂载进自己的工作区；分类模板通过 spawn 柔性分发影响 session 自改；飞书 wiki/多维表是下游只读镜像。scheduler 每奇数日 03:17 通过 /api/spawn 叫醒 FP 跑四阶段巡检。
怎么维护？
FP 只改自己 templates/ / rules/ / sop/ 下的文件。任何 session 的 CLAUDE.md / AGENTS.md 所有权都归 session 自己，FP 不强覆盖，只 spawn 通知；代改必须先拿确认。
怎么迭代？
主要走 pull 模式：git log 扫最近改动 + daily self-report + changelog 近 50 条反思。push 通道（/first-principle skill）保留但实际极少触发，已降级为 escape hatch。
有什么触发方式？
四种：user_command（用户直说）/ patrol（scheduler 定时）/ event（session 被纠正后自动提交）/ request（/first-principle skill push，legacy）。
维护哪些东西？
七类资产：3 份 Principles · 4 套 CLAUDE-md 分类模板（+ 对应 AGENTS-md 四套） · 3 份 rules（update-judgment / wiki-management / episodes） · 4 份 sop · 2 个脚本（sync-feishu / sync-session-table） · 1 个 changelog（SQLite + 飞书多维表镜像） · FP 自身 CLAUDE.md / AGENTS.md。
维护的东西怎么生效？
四种生效链路，看情况并用：symlink 瞬时（Principles） · spawn 柔性分发（分类模板） · 飞书 wiki 同步（11 份镜像给人类看） · changelog 留痕（SQLite + 飞书多维表镜像）。
如何保证 FP 自己不偏？
强制对称（FP 自己的 CLAUDE.md 和 AGENTS.md 互查，和 session 一套规则） · 每次编辑和巡检都必须问"还能删什么"（防 doc rot） · 多 session 交叉验证门槛（update-judgment v1.4/v1.5） · 所有变更走 changelog 留痕不允许静默改。

触发方式 — FP 的动作被什么叫起来

Command用户直接指令

  1. 用户直接说「改 Principles 的这条」或「做一次分类模板分发」
  2. FP 按相关 SOP 执行（distribution 或 operation-manual）
  3. 改 templates/ → commit → 同步飞书 wiki
  4. 如需下游执行（柔性分发），继续走 spawn 通道
  5. changelog 记 user_command

Cronscheduler 周期巡检

  1. 每奇数日 03:17，scheduler 通过 /api/spawn 调起 FP（prompt 指定"执行完整维护周期"）
  2. FP 按 sop/periodic-review-operation-manual.md 跑四阶段 Gather → Synthesize → Conform → Sync
  3. Phase 2 产生的内容变更直接改 templates/
  4. Phase 3 对齐违规 → spawn 通知 session 自改，记 patrol/deferred
  5. Phase 4 同步完，更新 .last-sync-review

Eventsession 被纠正后自动提交

  1. 某 session 被用户纠正 → 判断规则是"通用"还是"本 session 自用"
  2. 若通用 → 由该 session 调 /first-principle skill（落 requests/ pending 文件）
  3. FP 激活时扫 requests，按 rules/update-judgment.md 判断 5 种结果
  4. 处理后文件移入 requests/archive/
  5. changelog 记 event / request（二选一看来源）

legacy/first-principle skill push 通道

  1. push 通道名义上是上面 event 的落盘方式
  2. 现实：高强度使用多日以来极少成功触发 → 主信息源靠 pull（Phase 1 Gather）
  3. 保留作为 escape hatch，但 FP 不指望它

处理流程 — 一条请求进来，会落到哪 5 种结果

判断规则写在 rules/update-judgment.md（现 v1.5）。含两关：形式合法性 → 内容判断；多 session 同期出现同一问题会自动提升接受优先级。

accepted通过判断、写入 templates/ 或 rules/；commit + 飞书同步
rejected不符合原则 / 单 session 偏好 / 已在现有条款覆盖；仍留 changelog 记录拒绝原因
split一条请求混了多件事 — 拆分后分别判断
deferred单 session 首次提出、无 incident 证据、需要等交叉验证 — 暂缓
no_action巡检本身也要留痕 — 即使一切正常也写一行

scheduler 四阶段 — Gather → Synthesize → Conform → Sync

次序原则：先把内容定准，再查对齐。若先 Conform 再改内容，session 刚按旧模板改完又被打回。完整步骤见 sop/periodic-review-operation-manual.md。

Phase 1Gather — 收集本周期新信息
1.1 Code Change Review：对每个活跃工作区跑 git log --since，识别新 API / 新模式 / 踩坑 / 重复 revert。
1.2 Daily Self-Report：并发 spawn 活跃 session，让它们按 templates/daily-self-report-prompt.md 三段式回报（新能力 / 踩坑 / 跨 session 协作信号，每段 ≤100 字）。结果落 /tmp/fp-daily-selfreport-<date>/。
产物：Phase 2 的候选清单。
Phase 2Synthesize — 综合信号，更新内容
2.1 内容更新：把 Phase 1 候选转成 templates/*.md 的文字改动。多 session 投诉集中同一段 → 模板修订 + 判断是否重新分发。
2.2 Reflection：查 principles-log.db 近 50 条 → 多次 rejected / 多次 deferred / 多次 patrol fix 的模式 → 提炼进 rules/update-judgment.md。
2.3 Lint：三份 Principles + 四套分类模板互检（矛盾 / 失效引用 / 冗余）；各 session sop/ 目录健康检查。
2.4 剪枝（每阶段必做）：除了问"还缺什么"也问"什么过时了可以删"。
Phase 3Conform — 对齐 session 的 CLAUDE/AGENTS
对每个活跃 session 的 CLAUDE.md / AGENTS.md 做四项检查：
(a) 对称性 — CLAUDE 与 AGENTS 规则一致（仅 header 可说明后端不同）
(b) 分类模板符合性 — 与 templates/claude-md-{category}.md 对比
(c) BASE 残留 — grep BASE:BEGIN/END（已废除的机制，应清零）
(d) 语言合规（硬性）— 英文；中文仅限专有名词内嵌
违规 → spawn 通知 session 自改，不代改（代改需先 spawn 拿到确认）；记 patrol/deferred。
Phase 4Sync — 同步到下游
4.1 ./scripts/sync-session-table.sh → 飞书 session 多维表
4.2 ./scripts/sync-feishu.sh → 飞书 wiki（11 份：3 Principles + 4 分类模板 + claude-md-all-categories 合并视图 + rules/update-judgment + rules/wiki-management + sop/periodic-review-operation-manual）
4.3 date > .last-sync-review；无法自动处理的问题 → FP 群 lark-cli im +messages-send

生效链路 — 改完的东西怎么真正"起作用"

A. Symlink 瞬时 用于：3 份 Principles
每个 session 工作区里这 3 份文件是 指向 FP/templates/ 的软链。FP 一改即"全网可见"，无需推送，无需确认。这是 FP 影响力最直接的一条链。
B. Spawn 柔性分发 用于：4 套分类模板
FP 起草模板 → 先找用户确认"可以分发" → /api/spawn 并发通知同分类 session → 每个 session 按自身情况改自己的 CLAUDE.md/AGENTS.md → FP trust-but-verify（时间戳 / 语言 / 对称）。
C. 飞书 wiki 同步 用于：11 份给人看的镜像
sync-feishu.sh 把文件以 overwrite 模式 push 到 wiki。wiki 页只读，源在 git。给用户 / 其他 session / 外部人员浏览用。
D. Changelog 留痕 用于：所有变更
每一次 accepted / rejected / split / deferred / no_action 都要写 data/principles-log.db，并一行一行同步到飞书多维表。没写 changelog = 这次改没发生。

核心原则：柔性分发（Soft Distribution）

FP 从不强覆盖 session 的 CLAUDE.md / AGENTS.md

  • 所有权：每个 session 对自己的 CLAUDE.md / AGENTS.md 拥有所有权
  • 分类模板是 advisory：4 套 category 模板只是参考 demo，不是强制规则
  • 3 份 Principles 例外走 symlink：session 工作区里的 console/coding/business-principles.md 是软链到 first-principle/templates/，FP 改一次所有 session 立即生效——这是真正的单一事实源
  • BASE 强覆盖机制已废除：<!-- BASE:BEGIN/END --> 不再维护
  • 两层分发：
    • 主流程：FP spawn 通知 session，session 自主决定采纳或保留
    • FP 代改：必须先 spawn 取得 session 确认，不得静默改
  • Session 保留自身特色：业务特色、能力边界、工作区布局——session 自己填，FP 不干涉
  • 分类通用核心习惯 bottom-up：模板里的核心习惯都从该分类已有 session 实践抽共性
