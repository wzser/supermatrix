跨 Session 通讯机制盘点

SuperMatrix · /api/spawn + spawn-closure 三段闭环 · 代码出处 /src
盘点人：socail-king（SK）· 2026-05-19
一句话：一次通讯 = 一个结果。机器把一次 spawn 拆成「通讯执行」和「投递」两件事—— B 起来了、跑完了、回了话，只算通讯执行成功；A 真正要的那个东西落到了一个能被客观核对的地方，才算投递成功。 status=completed 只证明前者，不证明后者。这套机器存在的全部理由，就是不让前者冒充后者。
通讯执行（B 跑了、回了话） 投递（A 要的东西真落地了） 异步兜底 / 裁决

B 起来了吗
child session 进程是否成功启动。
✓ 判据：spawn 没报 spawn_failed

② 执行 EXECUTION
B 跑完且有产出吗
run 状态正常、final message 非空。
✓ 判据：非 timeout / 非 run_error / 产出 trim 后非空
失败 run_timeout / run_error / empty_output

✓ 判据：inbox 收到 / 表里有行 / 文件写了

↓
全过

200 closure:"verified"，把验证过的 finalMessage 返回 A。这才是真闭环。

通讯段失败

B 根本没起来 → 当场 500。通讯段的失败全在同步路了结，不进 watcher。

执行/投递失败

先复查一遍（防瞬态抖动）→ 仍不过 → 整条同步重试 B 一次（同 prompt）→ 再查。

run_timeout

特例：不做同步重试（大概率还超时），直接转异步。

↓
STEP 2 · 转异步
respondSwitchedAsync → registerAsyncItem
写一行到 spawn_async_items 表，返回 A：{ok:false, status:"switched_async", ref}。A 不再阻塞，之后 A 靠 GET /api/sessions/:id/result 轮询拿结果。
↓
STEP 3 · watcher 接管（SK 的 scheduler 任务）
定期扫 async 项 → D 类直接处理 / J 类升级裁决
D 类（纯路由，不需判断）：D1 = A 超时但 B 后来跑完且检查过 → 用 heartbeat 待办把结果送回 A；D2/D3/D4 = B 没起来/瞬态错/投递没发生 → redrive（程序化重发原 spawn，不走 heartbeat 消息）。
J 类（需判断）：重发用尽、反复空产出、结构性坏、约定反复改、卡过时长阈值 → spawn 一个 SK 裁决子 session，判是谁的锅、动手、把 verdict 写回 spawn_async_items。

3核心区分：「通讯执行」≠「投递」

这是整套机制的命门，也是 SK 这个 session 存在的理由。两者最容易被混为一谈——B 回了一长段话，看着像成了，但 A 要的东西可能根本没落地。

通讯执行

B 跑了、回了话

「A→B 的请求到达了，B 起得来，B 跑完了并吐出了非空内容。」——过程层面的成功。

投递

「A 当初要的那个具体结果，确实出现在了一个对话之外、能被第三方客观核对的地方。」——结果层面的成功。

投递是唯一拿对话之外的客观事实去对账的一段。A 不声明 delivery_checks，这一段就自动 pass——机器就退化成只验「B 回了话」。所以 A 必须把「算成功」声明成可检查的事实。

一句话记住差别：通讯执行问的是「B 回话了吗」，投递问的是「A 拿到它要的东西了吗」。 status=completed 只回答了第一个问题。SK 判定一次通讯成不成，永远看第二个。

4投递目的地：结果到底送去哪（resultSinks）

「投递」这个动作的物理落点由 resultSink 决定。sync_inline 的 child 由 HTTP handler 自己交付，sink 引擎让路；异步与派生场景才走引擎。出处 resultSinkEngine.ts。

sink 类型归类投递动作能否被 delivery_check 核对
http_response通讯执行结果随 /api/spawn 的同步 HTTP 响应原路返回 A。引擎里是 no-op，由 handler 交付。响应体本身即证据
pollable_endpoint通讯执行转异步后，结果落 message_runs，A 轮询 GET /api/sessions/:id/result 取。DB 行即证据
chat_post投递以 bot/user 身份把结果贴到某飞书群（父群 / 显式 chatId）。✓ inbox_message check
parent_continuation_inject投递合成 inbound 消息塞回父 session，唤醒父消费子的产出。父 busy/error/deleted 则丢弃。✓ 父 run 记录
eventbus_publish广播把固定 shape 的 payload 发到具名 topic，供订阅者消费。topic 投递，不入闭环
audit_only观察不做任何外发，只靠 cross_session_log 留痕。仅留档

5失败形态 × 在哪一段被发现 × 谁来兜

failureKind所在段含义处理路径
spawn_not_started① 通讯child 进程没起来。同步路当场 500，不进异步
run_timeout② 执行B 跑超过主 session 运行超时。不重试，直接转 async
run_error② 执行B 的 run 报错。复查 → 同步重试 1 次 → 仍错转 async（watcher D3 redrive）
empty_output② 执行B 的 final message 为空。同上；反复空 → J2 升级裁决
delivery_missing③ 投递声明的 delivery_check 没命中。同步重试 → watcher D4 redrive
late_result③ 投递A 已超时，B 后来才跑完。watcher D1：heartbeat 待办把结果送回 A

每个关键环节都写一条带 comm_id 的结构化闭环日志（closureLog.ts）：受理校验、三段每段判定、同步重试、转异步决定——任何一条 spawn 的全程都能事后 join 还原。

6别混淆：三层通讯的分工

协调层

HTTP API · /api/spawn

飞书 Feishu

观察层

EventBus / TopicBus

只做旁路观察、日志、通知。handler 里禁止改状态、起子 session。
