/api/spawn 三段检查 (快递员模型)

一条 spawn = 通讯(揽件) + 执行(打包) + 投递(送达)。快递员只判空不空, 不拆包裹看内容。

阶段判据失败形态failure_kind
通讯
Communication		 /api/spawn 调用成功, B 的 child session 创建成功 B 没起来 / child_session_id 为空 spawn_not_started
执行
Execution		 run status=completed + final_message 非空
例外: callerInvocation=fire_and_forget 跳过空产出
例外: 所有 sink 都是 audit_only → 空回合法		 run error / timeout / 产出为空 run_error / run_timeout / empty_output
投递
Delivery		 结果送到了声明的地址 (result_sink_attempts 有成功记录) 投递动作失败 / 无投递记录 delivery_missing / late_result

同步路径 (happy path)

  1. A 调 POST /api/spawn (from 必填), 阻塞等待
  2. 框架受理: 校验 from / sink 声明 / 节流检查 → 创 child B
  3. B 跑完 → 框架依次查三段
  4. 全过 → 返回 A 验证过的结果, 闭环
  5. 任一不过 → 框架同步重试 B 一次 (复查后重 spawn)
  6. 重试过 → 闭环; 还不过 → 登记 async, 返回 {ok:false, status:"switched_async"}
  7. 超时特例: 超时不做同步重试 (重试大概率也超时), 直接转 async

异步路径 (兜底)

  1. 框架登记 spawn_async_items 一行 (status=pending)
  2. A 收到 switched_async 回执, 不再阻塞
  3. Watcher 每 tick 扫描 pending 项 → 分流 D/J
  4. D 类: watcher 直接处理 (deliver 补送 / redrive 重发 / redeliver 重投)
  5. J 类: watcher spawn SK 裁决子 session
  6. SK 裁决 → 写 verdict → close async 项
  7. 补送一次即关项: 待办写入 heartbeat 池就关闭, 靠去重键防重复

Watcher 巡查流程

spawn-closure-watcher 是一个 scheduler cron 任务, 周期性扫描并处理 async 项。脚本位于 scripts/watcher-tick.sh, 通过 scheduler http executor 调 localhost:3501/api/spawn。

每个 tick: │ ├─ ① 孤儿扫描 (orphan sweep) [待实施] │ cross_session_log WHERE status='pending' AND child_session_id IS NULL │ AND created_at < now - 60s │ AND NOT EXISTS (spawn_async_items WHERE comm_id = id) │ → 登记 spawn_async_items (failure_kind=spawn_not_started) │ ├─ ② 扫 spawn_async_items WHERE status IN ('pending','re_driving','adjudicating') │ └─ ③ 对每条 async 项: 重新跑三段检查 → 分流 │ ├─ D 类 (直接处理, 无需判断) │ ├─ D1: A 超时 + B 后来跑完 + 结果通过检查 │ │ → 补送: 写 caller/A 的 heartbeat 待办 (带 B 完整结果) │ │ → 去重键: 当前 watcher 用 comm_id；05-19 历史/人工补投可见 comm_id:attempt │ │ → enqueue 前置: 目标 session 存在且 heartbeat_enabled=1 │ │ │ ├─ D2: B 没起来 → redrive (程序化重调 /api/spawn) │ ├─ D3: B 瞬态出错 → redrive │ └─ D4: 投递没发生但产出有效 → redeliver（重跑声明的 resultSinks, 不重跑 B） │ └─ J 类 (需要判断, 升级 SK) ├─ J1: delivery_missing + attempt 用尽 ├─ J2: empty_output + attempt 用尽 ├─ J3: run_error + attempt 用尽 ├─ J4: 其它未分类 + attempt 用尽 └─ J5: B 迟迟不完成 (stale/hung)
★ redrive 不走 heartbeat 消息。教训 (2026-05-19): heartbeat 把 "redrive_b" 口令注入 B, B 正确拒绝执行 —— 裸控制消息形态就是 prompt injection。
★ redrive = 程序化重调 /api/spawn (原 from/target/prompt + client_request_id=spawn-redrive:<logicalKey>), 新 spawn 走自己的三段检查。调用路径: watcher → scheduler http executor → localhost:3501/api/spawn。
★ D1 补送 = enqueue-heartbeat-todo CLI → heartbeat.sqlite session_todos 表, 去重键 = target_session + logical_key。enqueue 成功会关闭 watcher 项, 但不代表 heartbeat 已注入或目标 session 已处理。

耦合关系矩阵

谁依赖谁、怎么依赖、依赖的是什么

① 框架 → Scheduler

框架 spawn 失败 → 登记 spawn_async_items → Watcher (scheduler 任务) 接管

耦合点: spawn_async_items 表是框架写、watcher 读的共享状态。Watcher 的任务注册/ cron 调度依赖 scheduler 服务 (localhost:3500)。

紧耦合 — watcher 挂了, 所有 async 项静默积压

② Watcher → Heartbeat

Watcher D1 补送 → enqueue-heartbeat-todo CLI → session_todos 表

耦合点: session_todos 表 (heartbeat.sqlite)。Watcher 通过 CLI 写待办, heartbeat patrol 负责注入。去重靠 UNIQUE(target_session, logical_key), 且 CLI 先检查目标 session 是否存在、是否 heartbeat_enabled。

紧耦合 — 目标 session 未启用 heartbeat 时 enqueue 直接返回 target_not_heartbeat_enabled; 已入池后若 session 被 stop/permanent pause, 待办会积压在 drain（消费）侧而不是 watcher 写入失败 (当前 deepsearch 75 条 pending)

③ Watcher → SK

Watcher J 类 → spawn SK 裁决子 session → SK 写 verdict → close async

耦合点: Watcher 通过 /api/spawn 调起 SK 子 session, prompt 内含 async ref + comm 现场。SK 按 SOP 裁决后回写 spawn_async_items.verdict。

紧耦合 — SK 子 session 被限速/超时则 async 项停在 adjudicating

④ Heartbeat → 目标 Session

Heartbeat patrol → pause 检查 → packet → prefilter → controller 决策 → idle drain（空闲时消费）→ lark-cli user send

耦合点: 待办注入依赖目标 session 的 group_id、lark-cli im +messages-send --as user、框架 Feishu ingress, 以及目标 session 当时 idle 且 latest run 不是 running。patrol 不是简单扫 pending 就注入, 它先构造 session packet (recent_runs / recent_cross_session / pending_todos / policy), 再经过 deterministic prefilter（确定性预筛）和 controller JSON（控制器 JSON 决策）。

中度耦合 — busy 时只记 todo_skipped_session_busy 并保留 pending 下轮再试; 但真正发送失败或缺 group_id 会把 claimed 待办标 failed, 不会自动回到 pending。去重只能防重复注入, 不能保证目标 session 后续 run 一定成功。

⑤ Scheduler → Spawn 调用方

Scheduler 管理 cron 任务 → 各 session 的定时任务 (cross-session-review 等) → 这些任务调 /api/spawn

耦合点: Scheduler 不是 spawn 系统的一部分, 但它管理的任务 (cross-session-review) 自己也产生 spawn。这就产生"自指"问题: review 产生的 spawn 失败也会进 watcher, watcher 也会升级 SK 裁决。

中度耦合

⑥ 框架重启 → 孤儿 Comm

框架进程重启 → 在飞的 spawn 变成 pending+无 child → 孤儿

耦合点: 框架重启时不优雅退出, 不主动标记在飞的 spawn。孤儿扫描 (05-21 新增) 是兜底而非根治。DB 里现有 18 条历史孤儿。

紧耦合 + 已知缺口

⑦ 三段检查 → spawn_async_items

同步路径失败 → 登记 async → 异步路径接管

耦合点: failed_phase 和 failure_kind 是同步路径写入、异步路径读取的关键字段。写错 → watcher 分流错 → 动作错。

实证: EP-1 (sync_inline spawn 被误判 delivery_missing), EP-2 (late_result + executionPassed=false 无分支)

紧耦合 — 字段语义一致性是硬依赖

⑧ spawn_queue → 框架 childSession

节流入队 → child terminate 触发 dequeue → 起跑新 child

耦合点: spawn_queue 表和 childSession 生命周期互相驱动。队列调度正确性依赖 child terminate 路径必然触发 dequeue。

紧耦合 — 新机制, 尚未经充分实战验证

从 SK 裁决中沉淀的重复异常模式, 治"如何处理对"

EP-1
synthetic delivery_missing		 sync_inline spawn 的投递被误判 delivery_missing。result_sink 记的是 skipped (note: sync_inline handler owns delivery), 但 delivery 检查只认 status=delivered。 三段检查不认 sync_inline 的投递路径 false_alarm — 不重推/不补投/不通知 3 条 (ATP 测试桩 + SK 自测)
EP-2
late_result		 B 健康但慢, 超出同步窗口后被登记 late_result。watcher 的 classifyAsyncItem 对 late_result+executionPassed=false 无分支, punt SK。 ① classifyAsyncItem 缺分支 ② sync_inline 窗口关了后无 fallback 投递路径 false_alarm 或 late_result_delivered — SK D1 补投迟到结果 70+ 条 (deepsearch 真源头判别器 fan-out 为主, 也有 gongying/wuliu1 跨会话)
EP-3
notify-type empty_output		 Caller 把 /api/spawn 当通知用, prompt 写"无须回复", B 正确沉默, 框架判 empty_output。 框架缺 notify-only 通道, caller 选错通道 parked — 待框架加 fire-and-forget 通道后解封 70+ 条 storm (amzdata→dataquery 通知)
