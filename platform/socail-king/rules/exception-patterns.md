跟 `coordination-patterns.md`(治协作"如何写对")分工不同。

## EP-1:sync_inline spawn 上的 synthetic delivery_missing —— 判 false_alarm

**长什么样**

watcher 把一条 async 项当 J1 派来:`failed_phase=delivery`、`failure_kind=delivery_missing`、
`attempt_count` 到上限、reason 是 "attempt budget exhausted for delivery_missing"。
但收现场一看,底层那条 spawn 其实早就闭环了。

**怎么认出来(全是独立 artifact,不靠任何人自述)**

- `cross_session_log` 那条:`status=completed`,`final_message` 跟 prompt 期望字面一致;
  `message_runs.stream_log` 里 assistant 末条就是期望内容。通讯段+执行段都过了。
- `result_sink_attempts` 只有一行:`sink_kind=http_response`、`status=skipped`,
  note 写着 `skipped: sync_inline handler owns delivery`。
  —— 投递是 sync_inline 的 HTTP 响应当场完成的,A 同步就拿到了结果。
- `spawn_predicates` 对本 comm **无行** —— 没有注册的投递约定,
  这个 `delivery_missing` 不是某个写坏的真实约定判出来的。
- prompt 形态是测试桩:ping/pong、"闭环回路测试"、"one-shot test child"、
  编号式 token(`TC003B-done`、`ATP-CLOSURE-FAIL-...`),且常有同窗口同批 comm。

**根因**

framework 的 delivery 段检查对 sync_inline spawn 误判。sync_inline 的投递是 HTTP 响应,
`result_sink_attempts` 把这条 sink 记成 `skipped`(note: sync_inline handler owns delivery),
而 delivery 检查只认 `status=delivered`,看不到 delivered 就判 `delivery_missing`。
合成测试(ATP / socail-king 自测)再把这个 `failure_kind` 注进 `spawn_async_items`
驱动 watcher→SK(J1)裁决路径。底层回路实际已闭,没有真实的 A 在等结果落空。

**怎么处理**

- 分类 = **误报 / false_alarm**。非 B 的锅(B 精确照办),非约定的锅(无 predicate)。
- 不必访谈:坐实的证据全在独立 artifact,自述零增量,别动 spawn 预算。
- 回写 `spawn_async_items`:`verdict='false_alarm'`、`status='closed'`。
- 无重推、无 heartbeat、不升级用户、不通知 caller。

**给 framework 的建议(累计观察)**

1. delivery 检查应把 `result_sink_attempts.status=skipped` 且 note 含
   `sync_inline handler owns delivery` 视同投递成功,别判 `delivery_missing`。
2. 或给 `spawn_async_items` 加 synthetic/test 标记位,watcher 直接跳过这类测试桩,
   不惊动 SK 裁决。
3. 合成测试收尾统一用规范 verdict(`false_alarm`),别用 `test_artifact` 等
   非规范值污染 Step 5 的 verdict 计数。

**裁决记录**

- `async_2d54dd4a`(tx-spawn-20260518-001)—— ATP self-spawn e2e
- `async_c097aed5`(tx-spawn-20260518-002)—— ATP self-spawn e2e TC003B
- `async_63007a7b`(tx-spawn-20260519-001)—— socail-king 闭环回路自测 ping/pong

## EP-2:late_result —— 健康但慢的 spawn 被误判,判 false_alarm + 补投迟到结果

**长什么样**

watcher 把一条 async 项派来:`failed_phase=delivery`、`failure_kind=late_result`、
`attempt_count=0`、reason 是 `unhandled failure_kind: late_result`。
收现场一看,B 根本没失败——只是跑得比同步窗口长。

**怎么认出来(全是独立 artifact,不靠任何人自述)**

- `cross_session_log` 那条:最终 `status=completed`,`final_message` 是完整的期望产出;
  `message_runs` 对应行 `status=completed`、`error_message` 空。执行段实际过了。
- 时间线:`spawn_async_items.created_at` 卡在 comm 创建后某个固定窗口(实测见过 30s);
  comm 真实 `finished_at` 远在其后(实测 ~126s、~17min)。B 是慢,不是挂。
- `result_sink_attempts` 那条 `sink_kind=http_response`、`status=skipped`,
  note `skipped: sync_inline handler owns delivery` —— 但 sync_inline 的 HTTP 窗口
  早在转 late_result 时就关了,迟到结果没有投递路径,停在 `final_message`。
- `spawn_predicates` 对本 comm 无行 —— 无注册约定,不是约定坏。

**判 hung 前必查(tx-004 的教训)**

长链路 spawn(B 自己又 spawn 了下游)运行期间,父 child 会话行 `updated_at` 不刷新、
`token_usage` 要到完成才落行。**不能据 `updated_at` 冻结 + 零 token_usage + run 跑得久
就判 B 挂了去 redrive** —— 先扫同窗口有没有在跑的 `child_<B>` 下游会话,有则是慢不是挂。
late_result 默认倾向『再等』,redrive 只在确证 B 死(run `status=error/timeout`、确无下游活动)时才用。

**根因**

1. watcher `classifyAsyncItem` 对 `late_result` 只有一条分支
   `late_result + executionPassed → deliver_result`;`executionPassed=false`(B 还在跑)
   无分支,落末尾 `default adjudicate`,报 `unhandled failure_kind` punt 给 SK。
   可 `executionPassed=false` 恰恰是 late_result 的常态语义。
2. 同步窗口短,真实创意/多会话/长跑任务天然超窗,late_result 被常态化误升级。
3. `http_response` sink 让位 sync_inline 被记 `skipped`,但 sync_inline 窗口已关,
   迟到结果没有 fallback 投递路径。

**怎么处理**

- 分类 = **误报 / false_alarm**。非 B 的锅(B 慢但成功),非约定的锅(无 predicate)。
- **先查迟到结果送到 caller 没有**:扫 heartbeat `session_todos` 有没有
  `logical_key=<comm_id>:<attempt>`、`todo_type=spawn_closure` 的待办。
  - 没有 → 做 D1 补投:`enqueue-heartbeat-todo --session <caller> --key <comm_id>:0
    --todo-type spawn_closure`,message 跟 watcher D 类同格式
    (`Spawn closure follow-up for comm_id=... Action: deliver_result. late result is now present; deliver it to caller`)。
  - 已有(watcher D 类已投 / 同批兄弟已投)→ 不重复投。
- 回写 `spawn_async_items`:`status='closed'`;verdict 取值见下。
- 不重推(会产生重复长跑/重复子任务)、不升级用户、无 predicate 可 patch。

**verdict 取值(2026-05-19 校准)**:本文初版写 `false_alarm`,实操已分两档,按是否真做了补投选:

- SK 实际 enqueue 了补投待办(迟到结果此前没送到 caller)→ `late_result_delivered`。这是真实补救,不是"其实早闭了",`false_alarm` 不准确。已闭兄弟 `async_9115d60f`/`async_29f66376`/`async_9465ab11` 均此。
- 收现场发现迟到结果已被 watcher D 类或同批兄弟补投过、SK 无需再动 → `false_alarm`(如 `async_409ba2d6`)。

`late_result_delivered` 不在 SOP 规范 verdict 集(retrying/escalated/contract_fixed/false_alarm/parked)内 —— 快递员模型新增了"补送"动作却没配套 verdict 档。**待 SOP/framework 补齐**;在此之前以上述两分法保持一致,别引入第三种写法。

**给 framework 的建议(累计观察)**

1. `classifyAsyncItem` 给 `late_result + executionPassed=false` 补分支:
   未到 stale 阈值 → `noop` 继续等(下轮再复检),只有过 stale(J5)才升级。
   消掉一整类无谓的 SK 唤起。
2. 更根本:一个健康、仍 `running` 的慢 spawn 不该被登记成 delivery 失败 async 项。
   判 late_result 前先查子 `message_run.status`——`running` 且无 error → `in_progress`,
   不入 async、不惊动 SK。
3. `sync_inline owns delivery` 但其 HTTP 窗口已关时,迟到结果的
   `result_sink`(`http_response/skipped`)需要一条 fallback 投递路径,
   否则每条 late_result 都得靠 watcher D 类或 SK 补投。
